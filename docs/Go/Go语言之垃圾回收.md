# Go语言之垃圾回收



## 三色标记法

### 简单规则

1. 黑白灰三种颜色；
2. 黑对象代表自身存活，其指向的对象都标记完成；
3. 灰对象代表自身存活，指向对象还未被标记；
4. 白对象代表，未被标记到的对象，可能是垃圾对象；
5. 标记开始前，从根对象(全局对象、栈上的局部变量)置黑，将其所指向的对象置灰色；
6. 标记期间，从灰色对象出发，向其所指向的对象置灰，然后本身置黑；
7. 标记结束，白色就是不可达对象，进行清扫；

#### 缺点

1. 每次垃圾回收都需要STW，影响用户体验



## 并发垃圾回收

允许用户协程和GC协程并发运行，速度快，用户体验良好



### 缺点

### 1. 漏标问题

存在对象未被标记问题，导致误删，造成不可逆转问题

<img src="https://s2.loli.net/2024/04/08/h87RQn1qXe2LAUV.png" alt="image-20240408201815093" style="zoom: 67%;" />

1. B持有C的引用；
2. A扫描完成，置黑；B是灰色，未完成扫描；
3. 用户协程，B删除对于C的引用，A添加对于C的引用；
4. C无法被标记，然后被删除，造成问题；

### 2. 多标问题

<img src="https://s2.loli.net/2024/04/08/8rbKBUW6e3RChkZ.png" alt="image-20240408202023391" style="zoom:67%;" />

1. A持有对B的引用，A置黑，B灰；
2. A删除对于B的引用，B不会被删除；
3. 会在下一轮的时候，进行删除，浮动垃圾；



## 屏障机制

主要为了解决漏标问题，提出了屏障机制

### 1. 强弱三色不变式

漏标的主要存在原因是：

1. 黑色对象指向了白色对象；
   - **强三色不变式**：白色对象不能被黑色对象引用；
2. 灰色，白色对象删除了白色对象；
   - **弱三色不变式**：白色对象可以被黑色对象引用，但是要能通过灰色对象到达；
3. 1发生在2之前；



### 2. 插入写屏障

直接破坏1，因为1的问题主要出现在黑色引用了白色，导致白色无法被标记，然后被回收，出现了问题；

在黑色引用白色时候，出发屏障，将白色置为灰色，问题解决



### 3. 删除写屏障

直接破坏2，因为2的问题出现在，灰色删除对于白色的引用，导致白色不能被标记，容易被误删，出现了问题；

在灰色删除白色的时候，直接将该对象置为灰色，问题解决；



### 问题

1. 屏障无法作用于栈对象，因为栈的变化比较快，容易造成额外的资源开销；
2. 如果屏障无法作用于栈对象，那么就会出现上述的问题，在栈中出现漏标的问题；



### 4. 混合写屏障

1. GC开始之前，将栈为单位分批置黑；
2. GC期间，栈上的对象直接置黑；
3. 堆对象正常启用插入写屏障和删除写屏障；



## case探讨

主要讨论极端情况下，混合写屏障为什么能够一一解决问题？

### case 1

背景介绍：A在栈中，C是堆白色对象，B是堆中灰色对象，B指向C；

moment1：A建立对于C的引用；

moment2：B删除对于B的引用；

这个对应于上述说的漏标的1，2顺序；但是这个不会出现危险情况

- 虽然A建立对于C的引用本身应该执行插入写屏障的，但是由于A是栈对象，失效；
- B删除C的引用，会将C置灰；



### case 2

背景介绍：A(灰色)，B(黑色)是栈对象，C是堆对象；

moment1：B建立对于C的引用；

moment2：A删除对于C的引用；

这个对应于上述说的漏标的1，2顺序；但是这个不会出现危险情况

- 乍一看，删除写屏障和插入写屏障都失效了，但是不会出现漏标的危害；因为这种情况根本就不存在，因为B不可能存在C的引用；
- B本身持有C的引用，那么C已经是灰色了(x)
- B持有A的引用，通过A间接找到C，不可能，因为逃逸分析，B不可能持有对于另一个栈的引用；
- B同一个栈内的存在其他X能够到达C，X能够到达C，C不用被担心无法被标记；
