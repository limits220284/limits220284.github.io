四层网络设备：

- 应用层： 负责向用户提供一组应用程序，比如http、dns、ftp等；
- 传输层：负责端到端的通信，比如TCP、UDP等；
- 网络层：负责网络包的封装、分片、路由、转发，比如IP、ICMP等；
- 网络接口层：负责网络包在物理网络中的传输，比如网络包的封帧、MAC寻址、差错检测，以及通过网卡传输网络帧等；



## 域名解析的过程？

1. 先查浏览器缓存、到操作系统缓存、本地host文件；
2. 客户端首先会发出一个 DNS 请求，问 www.server.com 的 IP 是啥，并发给本地 DNS 服务器（也就是客户端的 TCP/IP 设置中填写的 DNS 服务器地址）。
3. 本地域名服务器收到客户端的请求后，如果缓存里的表格能找到 www.server.com，则它直接返回 IP 地址。如果没有，本地 DNS 会去问它的根域名服务器： **根域名服务器是最高层次的，它不直接用于域名解析，但能指明一条道路**。
4. 根 DNS 收到来自本地 DNS 的请求后，发现后置是 .com，给本地DNS服务器.com的服务器地址；
5. 本地 DNS 收到顶级域名服务器的地址后，发起请求问 www.server.com 的 IP 地址；
6. 顶级域名服务器给出权威域名服务器的地址；
7. 本地 DNS 于是转向问权威 DNS 服务器、server.com 的权威 DNS 服务器，它是域名解析结果的原出处。
8. 权威 DNS 服务器查询后将对应的 IP 地址 X.X.X.X 告诉本地 DNS。
9. 本地 DNS 再将 IP 地址返回客户端，客户端和目标建立连接。



## 键入网址到网页显示，期间发生了什么？

- 浏览器的第一步工作就是对URL协议进行解析，生成请求信息；

- 查询服务器域名对应的IP地址；具体过程就是先请求根域名，然后顶级域名，到权威域名服务器；

- 进行TCP连接，三次握手等，生成**数据段**；

- TCP连接生成的数据段需要IP模块将其发送出去，通过IP进行远程定位，生成IP**数据包**；

- 网络中传输，到了数据链路层部分，此时需要mac地址定位具体的主机在哪，生成**数据帧**，MAC地址通过ARP地址解析协议获取，其中涉及到**ARP缓存**；

- 然后数据从网卡出去，交换机送出即可；

  - 交换机是二层网络设备，端口不具有MAC地址，但是存在MAC地址表，包含设备MAC地址和对应的端口；
  - 如果不存在对应的映射，就会广播到除了源端口之外的所有端口；

- 网络包经过交换机，到达路由器，然后路由器根据ip地址和网关的映射，转发到下一个路由器；

  - 通过ip地址和子网掩码进行异或，然后和网关地址对比，然后决定路由地址；

- 到达对方之后，反向去除头部获取信息，最终送达服务端的某个服务即可；

## session  存在于哪里？说说cookie 和 token？

https://sloving.top/blogs/Note/chushiJWT.html

1. session存在于服务器；session 是另一种记录服务器和客户端会话状态的机制；

2. cookie是服务器发送给浏览器的小块数据，里面可能包含session_id，浏览器下次再次访问服务器时，如果在一个域下，就会自动携带cookie，做一些登录校验，信息共享等功能；分为会话cookie和持久cookie；
3. token是一种凭证，访问对应的资源接口；一般由 user_id + time + (权限范围) + (签名)组成；一般是用户通过账号密码登录后，服务器端验证无问题生成的，返回到浏览器，存储在cookie或者localstorage中；
4. jwt: token的一种，header + payload + signature组成，相比token，访问服务器，验证过程，获取用户信息等过程更加方便，而且能够解决跨域，csrf攻击；

## web接口比较慢，有哪些方式进行排查？

#### 网络

1. 检查网络：通过ping或者traceroute命令检查网络连接状态
2. 资源加载：是否存在大量的前端资源加载问题

#### 数据库

1. 慢查询优化：开启数据库的慢查询日志功能，分析执行时间长的SQL查询，并进行优化。
2. 索引优化：是否对常用的key进行建立索引，或者查询是否走了索引? 采用explain命令查看详细信息
3. 优化sql查询：比如select * 这种尽量不要存在

#### 服务器配置资源

1. CPU和内存是否过大，是否存在资源的限制，通过TOP可以查询，P是按照CPU占用率倒序，M是内存倒序

#### 应用程序

1. 代码程序复杂度是否过高，可以通过一些代码分析工具分析哪些代码执行时间长；
2. 是否存在并发逻辑问题；



## 没有建立连接，客户端直接发送数据给客户端会发生什么？


如果没有建立连接，客户端尝试直接发送数据给服务器通常会导致以下几种情况，具体取决于所使用的通信协议：

### 使用TCP协议

TCP（传输控制协议）是一种面向连接的协议，要求在数据传输之前必须建立连接。如果尝试在没有建立TCP连接的情况下发送数据：

- **连接未建立**：客户端无法直接发送数据包给服务器，因为TCP协议要求在数据传输前通过三次握手过程建立连接。如果客户端尝试发送数据，操作系统通常会返回一个错误，提示没有建立连接。
- **数据丢失**：即使数据包被客户端强行发送出去，由于服务器端没有与之对应的连接状态，服务器将无法正确处理这些数据包，数据包会被服务器忽略或丢弃。

### 使用UDP协议

UDP（用户数据报协议）是一种无连接的协议，它允许数据包（数据报）在没有预先建立连接的情况下被发送。

- **数据可能到达**：使用UDP，客户端可以在没有建立连接的情况下向服务器发送数据包。因为UDP不要求建立连接，数据包有可能到达服务器。
- **无可靠性保证**：UDP不保证数据包的到达顺序、可靠性或数据完整性。发送的数据可能会丢失、重复或乱序到达，而且UDP协议本身不提供重传机制或到达确认。





## 如果服务端向客户端发了多次请求得不到回应，断开连接之后，客户端再向服务端发请求，会收到怎样一个信息？

TCP:  收到RST报文

UDP: (当然这里已经明确指出了是断开连接了，所以应该是TCP连接)，如果是UDP的话，应该是没有响应，或者响应数据；





## 四次挥手的目的

1. 双方都可以主动的关闭连接
2. 数据完整性和可靠性
3. 防止旧的连接数据包干扰新连接





## 操作系统，有哪些具体服务可能是UDP的？

1. DNS: 域名解析协议
2. SNMP: 简单网络管理协议
3. TFTP: 简单文件传输协议
4. RTP: 实时传输协议
