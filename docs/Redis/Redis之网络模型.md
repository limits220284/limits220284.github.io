# Redis之网络模型

![image-20231221151903863](https://s2.loli.net/2023/12/21/I1NXQDFEPOn8s3K.png)

Linux系统为了提高IO效率，会在用户空间和内核空间都加入缓冲区

- 写数据的时候，要把用户缓冲数据拷贝到内核缓冲区，然后写入设备
- 读数据时，要从设备读取数据到缓冲区中，然后拷贝到用户缓冲区

![image-20231221152121323](https://s2.loli.net/2023/12/21/SemyqB9K8Rwdzca.png)

## 阻塞IO

顾名思义，阻塞IO就是两个阶段都必须阻塞等待

![image-20231221152337829](https://s2.loli.net/2023/12/21/YQEiWu5IpoFdRtT.png)

## 非阻塞IO

顾名思义，非阻塞IO的recvfrom操作会立即返回结果而不是阻塞用户进程。

![image-20231221152530303](https://s2.loli.net/2023/12/21/AkKgp6S9oBZFjVL.png)

可以看到，非阻塞IO模型中，用户进程在第一个阶段是非阻塞，第二个阶段是阻塞状态。虽然是非阻塞，但是性能并没有得到提高。而忙等机制会导致cpu空转，cpu使用率暴增。

## IO多路复用

无论是阻塞还是非阻塞IO，用户应用在一阶段都需要调用recvfrom来获取数据，差别在于无数据时的处理方案：

- 如果调用recvfrom时，恰好没有数据，阻塞IO会使进程阻塞，非阻塞IO使CPU空转，都不能充分发挥作用。
- 如果调用recvfrom时，恰好有数据，则用户进程可以直接进入第二阶段，读取并处理数据。

比如服务端处理客户端socket请求时，在单线程的情况下，只能依次处理每一个socket，如果正在处理的socket恰好未就绪(数据不可读或者不可写)，线程就会被阻塞，所有其他客户端socket都必须等待，性能自然会很差。

### 解决方案

1. 多线程处理
2. 哪个任务的数据就绪了，就先处理哪个的

### 文件描述符

文件描述符(File Descriptor)：简称FD，是一个从0开始递增的无符号整数，用来关联Linux中的一个文件。在linux中，一切皆文件，，例如常规文件，视频，设备等，当然也包括网络套接字。

**IO多路复用**：是利用单个线程同时监听多个FD，并在某个FD可读、可写时得到通知，从而避免无效等待，充分利用cpu资源

![image-20231221154218165](https://s2.loli.net/2023/12/21/ONy1MsQrxELefSl.png)

### IO多路复用的监听方式

- select
- poll
- epoll

### select

![image-20231221155128778](https://s2.loli.net/2023/12/21/6RIVwxPyNTHijEF.png)

#### 存在的问题

- 需要将整个fd_set从用户空间拷贝到内核空间，select结束还要再次拷贝回用户空间，繁琐
- select无法得知具体是哪个fd值，只能遍历，时间复杂度高
- fd_set监听的fd数量不超过1024

#### POLL

IO流程

- 创建pollfd数组，向其中添加关注的fd信息，数组大小自定义
- 调用poll函数，将pollfd数组拷贝到内核空间中，转链表存储，无上限
- 内核遍历fd，判断是否就绪
- 数据就绪或者超时后，拷贝pollfd数据到用户空间，返回就绪fd数量n
- 用户进程判断n是否大于零，然后遍历pollfd数组，找到就绪的fd

![image-20231222024526236](https://s2.loli.net/2023/12/22/gKoFU47NPcrqBad.png)

#### 存在的问题

- select模式中的fd_set大小固定是1024，而pollfd在内核中采用链表，理论无上限
- 监听fd越多，每次遍历消耗时间也越久，性能反而会下降

### epoll

epoll模式是对select和poll的改进，它提供三个函数:

![image-20231222023734241](https://s2.loli.net/2023/12/22/iGNWbTsBjEQ3cXl.png)

#### 优势

- 基于红黑树保存要监听的FD，理论上是没有上限，而且增删改查的效率都很高，性能不会随着监听的FD数量增多而下降；
- 内核会将就绪的FD直接拷贝到用户空间的指定位置，用户进程无需遍历所有的FD就能知道FD是谁；

#### 事件通知机制

当FD有数据可读时，我们调用epoll_wait就可以得到通知。但是事件通知的模式有两种:

- LevelTriggered：简称LT。当FD有数据可读时，会重复通知多次，直到数据处理完成。是Epoll的默认模式
  - 重复通知，效率存在问题
  - 多线程存在竞争问题
- EdgeTriggered：简称ET。当FD有数据可读时，只会被通知一次，不管数据是否处理完成。
  - 容易造成数据没读完，FD就被删除了
  - 结合非阻塞IO，效率会更好一点

举个例子:

1. 假设一个客户端socket对应的FD已经注册到了epoll实例中
2. 客户端socket发送了2kb的数据
3. 服务端调用epoll_wait，得到通知说FD就绪
4. 服务端从FD读取了1kb数据
5. 回到步骤3(再次调用epoll_wait，形成循环)

## 信号驱动IO

**信号驱动IO**是与内核简历SIGIO信号关联并设置回调，当内核有FD就绪的时，就会发送SIGIO信号通知用户，期间用户以执行其他业务，无需阻塞等待。

![image-20231223143031766](https://s2.loli.net/2023/12/23/EFxIz98sMplJbOh.png)

### 缺点

- 当有大量IO操作的时，信号较多，SIGIO处理函数不能及时处理可能导致信号队列溢出
- 当内核空间与用户空间的频繁信号交互，性能比较低

### 异步IO

异步IO的整个过程都是非阻塞的，用户进程调用完异步API后就可以去做其他事情，内核等待数据就绪并拷贝到用户空间才会递交信号，通知用户进程。

![image-20231223144418499](https://s2.loli.net/2023/12/23/2q4YDPjpVE9az6v.png)

可以看到异步IO的两个过程都是非阻塞的

### Redis网络模型

#### Redis到底是单线程还是多线程？

- 如果仅仅聊Redis的核心业务部分(命令处理)，答案是单线程
- 如果是聊整个Redis，那么答案是多线程

在Redis版本迭代的过程中，在两个重要的时间节点上引入了多线程的支持:

- Redis v4.0：引入多线程异步处理一些耗时比较长的任务，例如异步删除命令unlink
- Redis v6.0：在核心网络模型中引入多线程，进一步提高对于多核CPU的利用率

#### 为什么Redis要选择单线程？

- 抛开持久化不谈，**Redis是纯内存操作，执行速度非常快，它的性能瓶颈是网络延迟而不是执行速度**，因此多线程并不会带来巨大的性能提升。
- 多线程会导致过多的上下文切换，带来不必要的开销
- 引入多线程会面临线程安全问题，必然要引入线程锁这样的安全手段，实现复杂度增高，而且性能会大打折扣
