# 锁

## 数据库有哪些锁？

### 全局锁

> 为什么要加全局锁？

1. 防止多表更新导致的表之间的数据不一致，如果此时进行备份操作，会出现错误；
   - 可重复读隔离级别也可以解决这个问题，前提是引擎支持事务；

> 全局锁怎么用的？

```sql
flush tables with read lock -- 锁住的是整个数据库
```

insert、delete、update、alter table、drop table会发生阻塞；

释放时机：

1. unlock tables
2. 会话断开

### 表级锁

#### 表锁

```sql
lock tables t_student read; -- 表级别共享锁
lock tables t_student write; -- 标记表独占锁
```

**Tips：表锁除了会限制别的线程的读写，也会限制接下来本线程的读写操作；**



#### 元数据锁

不需要显示的使用MDL，当我们对数据库进行操作的时候，会自动的给这个表加上MDL：

- 对一张表进行CRUD操作时，加的是MDL读锁；
- 对一张表做结构变更操作的时候，加的是MDL写锁；

> 为什么需要元数据锁？

防止当前线程对这个表进行crud的时候，别的线程对这个表进行了表结构变更；

> MDL不需要显示调用，那它是在什么时候释放的？

MDL是在事务提交时候才会释放的，这意味着事务执行期间，MDL是一直持有的；

> MDL存在什么风险?

**数据库线程爆满风险**：

场景：A事务执行crud，获取了MDL读锁，但是持久未提交，B事务要进行表结构更新，阻塞ing；

后续所有操作全部阻塞，导致数据库线程爆满；

> 为什么B事务申请不到MDL写锁，而导致后续的申请读锁的查询操作也会被阻塞？

因为申请MDL锁的操作会形成一个队列，队列中写锁获取优先级高于读锁；



#### 意向锁

> 意向锁分为哪几类？

1. 意向共享锁(行级记录需要加共享锁前提)
2. 意向独占锁(行级记录需要加独占锁前提)

> 为什么存在意向锁？

问题主要在于如果需要对表进行 行数据的操作，如果不加意向锁，此时再加独占表锁的时候，就需要一行行的遍历所有记录，查询记录是否存在独占锁，效率很慢；

> 意向锁怎么解决的上述问题？

在执行插入、更新、删除操作，需要对表加上意向独占锁，然后再对该记录加锁；

意向共享锁和意向独占锁是表级锁，不会和行级锁发生冲突，但是会和共享表锁和独占表锁发生冲突。

**那么如果加入了意向共享锁，如果有需要加独占表锁的时候，就会检查是否存在意向独占锁，从而快速判断；**

### 行级锁

select语句不会对记录进行加锁，因为它属于快照读；如果要在查询时候对记录进行加锁，可以使用下面这两种方式，这种查询会加锁的语句成为**锁定读**；

```sql
//对读取的记录加共享锁
select ... lock in share mode;

//对读取的记录加独占锁
select ... for update;
```

<img src="https://cdn.xiaolincoding.com/gh/xiaolincoder/mysql/%E9%94%81/x%E9%94%81%E5%92%8Cs%E9%94%81.png" alt="img" style="zoom:67%;" />

行级锁主要包括三个：

1. 记录锁(分为X和S，X和S不兼容)
2. 间隙锁(间隙锁分为X和S，但是X和X是可以兼容的)
3. 临键锁(分为X和S，X和S不兼容，主要原因是临键锁包括了记录锁)

#### 插入意向锁

一个事务想要插入某一条记录的时候，需要判断该位置是否已经被其他事务加入了间隙锁，如果有的话，插入操作就会发生阻塞，直到拥有那个间隙锁的事务提交为止，在此期间会形成一个**插入意向锁**；

> mysql加锁的时候，先生成锁结构，然后设置锁的状态，如果锁状态是等待状态，并不是意味着事务获取到了锁，只有锁为正常状态时候，才意味着事务获取了锁的状态；

**插入意向锁**是行级锁，但是并不是意向锁，一种特殊的间隙锁，**锁住的是某一个点**；

> 那插入意向锁解决了什么问题呢？

1. 锁升级问题：其实插入意向锁并不直接导致锁升级，而是更好的帮助DBMS进行锁升级；
   - **锁升级**（Lock Escalation）是数据库管理系统（DBMS）中一种常见的锁管理策略，用于处理和优化锁资源的使用。锁升级涉及将较低级别的锁（例如行锁）自动转换为更高级别的锁（例如页锁或表锁）。这个机制主要目的是减少锁的数量，从而减轻锁管理的开销，提高系统性能。
     - **行锁超过阈值**：如果某个事务在单表中锁定的行数达到一定的阈值，就会触发锁升级；
     - **资源紧张**：如果内存紧张，可能也会采用锁升级；
2. 提高并发性
3. 避免死锁
   - 由于锁之间的关系更加清楚，减少了死锁发生的概率



## 什么SQL语句会加行级锁？

普通的select语句是不会对记录进行加锁的(除了串行化隔离级别)，因为它属于快照读，通过MVCC实现的

如果要在查询的时候对记录加行级锁，可以使用下面这两种方式，这两种查询会加锁的语句成为**锁定读**

```sql
// 对读取的记录加共享锁(S型锁)
select ... lock in share mode;
// 对读取的记录加独占锁(X型锁)
select ... for update;
```



除了上面这两条锁定读语句会加行级锁之外，**update和delete操作都会加行级锁**，且锁的类型都是独占锁(X型锁)；

```sql
//对操作的记录加独占锁(X型锁)
update table .... where id = 1;

//对操作的记录加独占锁(X型锁)
delete from table where id = 1;
```

## 行级锁的种类

**不同的隔离级别下，行级锁的种类是不同的。**



**在读已提交隔离级别下，行级锁的种类只有记录锁，也就是仅仅把一条记录锁上；**



在**可重复读隔离级别**下，行级锁除了有记录锁，还有间隙锁(目的是避免幻读)，所以行级锁的种类主要有三类:

- Record Lock，记录锁，也就是把一条记录锁上；
- Gap Lock，间隙锁，锁定一个范围，但是不包含记录本身；
- Next-key Lock： Record Lock + Gap Lock的组合，锁定一个范围，并且锁定记录本身



### Record Lock

```sql
begin;
select * from t_test where id = 1 for update;
```

事务会对id=1的记录上一把X型锁(行级独占锁)，其他事务如果想进行更新和删除都是会被阻塞的；如果其他事务想插入一条数据，那么不会被阻塞，而是报主键冲突的错误(因为主键有唯一性约束)；

当事务commit之后，事务过程中生成的锁都会被释放；



### Gap Lock

间隙锁，只存在可重复读隔离级别，目的是为了解决可重复读隔离级别下的幻读现象；

假设，表中有一个范围 id 为（3，5）间隙锁，那么其他事务就无法插入 id = 4 这条记录了，这样就有效的防止幻读现象的发生。

![img](https://s2.loli.net/2024/04/16/vmajTntiHwJ8QF4.png)

**tip:** 间隙锁虽然存在X型间隙锁和S型间隙锁，但是没有什么区别，间隙锁之间是兼容的，即两个事务可以同时持有包含共同间隙范围的间隙锁，并不存在互斥关系，因为间隙锁的目的是为了防止插入幻读记录而提出的；



### Next-key Lock

Next-Key Lock 称为临键锁，是 Record Lock + Gap Lock 的组合，锁定一个范围，并且锁定记录本身。

假设，表中有一个范围 id 为（3，5] 的 next-key lock，那么其他事务即不能插入 id = 4 记录，也不能修改和删除 id = 5 这条记录。

![img](https://s2.loli.net/2024/04/16/v1cSoZPMNpF4Xhd.png)

所以，next-key lock 即能保护该记录，又能阻止其他事务将新记录插入到被保护记录前面的间隙中。

**tips:** next-key-lock是包含间隙锁和记录锁的，如果一个事务获取了X型的next-key lock，那么另一个事务在获取相同范围的X型next-key lock时，是会被阻塞的，就是因为存在记录锁，所以无法同时获取记录的更改权；



## MySQL是怎么加行级锁的？

加锁的对象是索引，加锁的基本单位是next-key lock；

next-key lock在一定程度上会退化成记录锁或者是间隙锁

- 在能够使用记录所和间隙锁就能够避免幻读现象的场景下，next-key lock就会退化成记录锁或间隙锁

### 唯一索引等值查询

#### 记录存在的情况下

可以通过

```sql
 select * from performance_schema.data_locks\G;
```

<img src="https://s2.loli.net/2024/04/16/yxbC9QqSPXRNpvU.png" alt="image-20240416140839377" style="zoom: 80%;" />



从上图可以看到，共加了两把锁，分别是:

- 表锁：X类型的意向锁；
- 行锁：X类型的记录锁；
  - 如果LOCK_MODE是**X**，说明是**next-key lock**
  - 如果LOCK_MODE是**X，REC_NOT_GAP**，说明是记录锁；
  - 如果LOCK_MODE是**X，GAP**，说明是**间隙锁**；

**加锁的对象针对的是索引**

> 为什么上述情况下，唯一索引等值查询并且查询记录存在的情况下，该记录的索引中next-key lock会退化成记录锁？

ans: 能够避免幻读现象

why: 幻读的定义是当一个事务前后两次查询的结果集，不相同就认为了发生了幻读，所以要避免幻读就是避免结果集某一条记录被其他事务删除，或者有其他事务插入了一条新的记录，这样前后两次查询的结果就会不一样。这次实验里只针对了id=1的记录，也就是结果集只有这一条记录，只要保证它不被删除或者被更新即可；

#### 记录不存在的情况下

```sql
mysql> begin;
Query OK, 0 rows affected (0.00 sec)

mysql> select * from user where id = 2 for update;
Empty set (0.03 sec)
```

<img src="https://s2.loli.net/2024/04/16/NtJi35avRpD4EWf.png" alt="image-20240416141939773" style="zoom:80%;" />



从上图看，加了两个锁，分别是：

- 表锁，X类型意向锁；
- 行锁，X类型间隙锁；



因此，此时事务在id=5记录的主键索引上加的是间隙锁，锁住的范围是(1, 5)

![img](https://s2.loli.net/2024/04/16/wUFIic1MLln6obQ.png)

接下来如果存在其他事务插入id值为2，3，4这一些记录的话，这些插入语句就会发生阻塞；为什么不是next-key lock呢？因为id=1和id=5的记录已经存在，通过主键唯一性就可以保证；

> 可能会好奇，为什么能够确定(1, 5)是最终的间隙范围呢？

如果LOCK_MODE是next-key或者间隙锁，那么LOCK_DATA就表示锁的范围[右边界]，此次事务A的LOCK_DATA就是5；

然后锁范围左边界是表中id=5的上一条记录的id值，即1，所以间隙锁的范围是(1, 5);



>  为什么唯一索引等值查询并且查询记录「不存在」的场景下，在索引树找到第一条大于该查询记录的记录后，要将该记录的索引中的 next-key lock 会退化成「间隙锁」？

总体来说，还是因为间隙锁就已经能够避免幻读问题？

从结果集上考虑，如果当前查id=2不存在，那么应该怎么做才能避免下次查询不会出现别的事务插入id=2的记录的情况

- 在这个间隙范围内上一把锁，避免别的事务进行插入即可；
- id=1 or id=5，通过主键唯一性即可避免插入；
- 无需给id=5上记录锁，因为当前查询是id=2的结果，删不删除id=5的记录对最后的结果影响不大；
- 为什么不可以针对不存在的记录加记录锁？因为锁是加在索引上的，这个场景下，查询的记录是不存在的，所以自然没办法锁住这条不存在的记录；



### 唯一索引范围查询

当唯一索引进行范围查询时，**会对每一个扫描到的索引加 next-key 锁，然后如果遇到下面这些情况，会退化成记录锁或者间隙锁**：

- 针对大于等于，如果等值查询存在于表中，那么该记录的索引中的next-key锁会退化成**记录锁**；

- 针对小于或小于等于，要看条件值是否存在于表中：
  - 当条件值不存在，不论小于还是小于等于，next-key都会退化成间隙锁；
  - 当条件值存在，对于小于，退化成间隙锁，对于小于等于，保持next-key lock；
  - 对于扫描到的记录，都加next-key lock；



#### 实验一：针对大于情况

```sql
select * from user where id > 15 for update;
```



<img src="https://s2.loli.net/2024/05/05/1jXNAklizw5HLuC.png" alt="img" style="zoom: 67%;" />



分析：

- 在id=20这条记录的主键索引上，加了范围为(15, 20]的next-key锁，其他事务无法插入[16,19]的记录，也无法修改20这条记录；
- 在特殊记录(supermum preudo-record)的主键索引上，加了范围为(20, + $\infty]$ 的next-key lock；



#### 实验二：针对大于等于的范围查询情况

```sql
select * from user where id >= 15 for update;
```

<img src="https://s2.loli.net/2024/05/05/Blw5rQEYTpoWDGF.png" alt="img" style="zoom:67%;" />

分析：

- id=15是等值查询，next-key锁会退化成记录锁，仅锁住id=15这一行；
- 由于是范围查找，基会继续往后找存在的记录，扫描到第二行是id=20，于是对该主键索引加的是范围为(15,20]的next-key lock；
- 然后对于id=20之后的加入的是(20, + $\infty]$的next-key lock；

#### 实验三：针对小于情况

```sql
select * from user where id < 6 for update;
```

<img src="https://s2.loli.net/2024/05/05/27no1qzLTJIr6DY.png" alt="img" style="zoom:67%;" />

分析：

- 最开始找的是id=1，于是对该主键索引加的是范围(-$\infty$, 1]的next-key lock；
- 然后往后遍历，加(1，5] next-key lock；
- 第三行是id=10，不满足<6，退化成间隙锁，加(5,10)的间隙锁；

```sql
select * from user where id <= 5 for update;
```

- 然后往后遍历，加(1，5] next-key lock；
- 结束；
