## 如何避免索引失效？

### 失效的情况

1. 对索引使用左模糊匹配或者左右模糊匹配
2. 对索引使用函数
3. 对索引进行表达式计算
4. 对索引隐式类型转换
5. 联合索引非最左匹配
6. where子句中的OR

### 题目一

2. 右匹配，仍然会走索引扫描

### 题目二

1. 走索引扫描
2. 索引扫描，但**覆盖索引**
3. 索引扫描，但**覆盖索引**
4. 索引扫描，但**覆盖索引**

为什么会出现这种情况呢？

首先这张表的字段没有非索引字段，所以select*等价于select id，name；然后这个查询的数据都在二级索引的B+树中，**因为二级索引B+树的叶子节点包含索引值和主键值，所以查二级索引的B+树就能够查到全部的结果了，这个就是覆盖索引。**

#### 为什么选择全扫描二级索引树，而不扫描聚簇索引树呢？

因为二级索引树记录的东西很少，只有[索引列+主键值]，而聚簇索引记录的东西会更多，比如聚簇索引中的叶子节点则记录了主键值、事务id、用于事务的MVCC的回滚指针以及所有的剩余列。

再加上，这个select*不用执行回表操作。

所以MySql优化器认为直接遍历二级索引树要比遍历聚簇索引树的成本小的多，因此MySql选择了[全扫描二级索引树]的方式查询数据；

### count()是什么？

count()是一个聚合函数，函数的参数不仅可以是字段名，也可以是其他任意表达式，该函数的作用是统计符合查询条件的记录中，函数指定的参数不为null的记录有多少个。

假设count()函数参数是字段名name，意思就是统计该表中所有name不为空的记录有多少个。

再来假设 count() 函数的参数是数字 1 这个表达式，如下：

```sql
select count(1) from t_order;
```

这条语句是统计「 t_order 表中，1 这个表达式不为 NULL 的记录」有多少个。

1 这个表达式就是单纯数字，它永远都不是 NULL，所以上面这条语句，其实是在统计 t_order 表中有多少个记录

1. count(主键字段)会优先选择二级索引然后再选择聚簇索引(主键索引)，前者维护的信息更少一点，但都是通过遍历得到最后的结果；
2. count(1)，如果只有聚簇索引，那么就遍历聚簇索引，将读取到的记录返回给server层，但是server层不会读取记录，读到一条count就+1，所以比count(主键)快；
3. count(*)，等于count(0)和count(1)一致；但是MySql会对count(\*) 和count(1)有优化，如果有多个二级索引时，优化器会使用key_len最小的二级索引进行扫描；只有当没有二级索引的时候，才会使用主键索引进行统计；
4. count(字段)，使用全表扫描；

## 为什么要通过遍历的方式来计数？

因为InnoDB存储引擎是支持事务的，同一时刻多个查询，由于多版本并发控制(MVCC)的原因，InnoDB表应该返回多少行也是不确定的；

## 除了B+树，还有什么索引结构？比如ES用的是什么索引？

Elasticsearch（ES）使用的是一种名为倒排索引（Inverted Index）的数据结构，专门针对全文搜索进行了优化；

### 1. 倒排索引（Inverted Index）

- **特点**：倒排索引是一种将文档中的数据映射到包含这些数据的文档列表的结构。主要用于全文搜索引擎，如Elasticsearch和Apache Lucene。
- **适用场景**：非常适合于快速全文搜索，因为它允许以极高效率查找包含特定词汇的所有文档。
- **实现方式**：通常包括两个主要组成部分——一个是唯一词汇的列表，另一个是每个词汇对应的文档列表。

### 2. Trie树（前缀树）

- **特点**：Trie树，也称为前缀树或字典树，是一种用于快速检索字符串集合中某个字符串的树形数据结构。
- **适用场景**：适用于自动补全、拼写检查、IP路由查找等需要快速前缀匹配的场景。
- **实现方式**：树中的每个节点代表字符串中的一个字符，从根节点到某一节点的路径表示一个字符串的前缀。



## 排查慢SQL查询是数据库性能优化的一个重要方面，可以通过以下步骤进行：

### 1. 启用慢查询日志

大多数数据库管理系统（如MySQL、PostgreSQL）都提供了慢查询日志功能，可以记录执行时间超过某个阈值的SQL语句。首先确保慢查询日志被启用并设置合适的阈值。

- **MySQL示例**：可以通过设置`slow_query_log`和`long_query_time`参数来启用慢查询日志并定义什么样的查询算作慢查询。

### 2. 分析慢查询日志

利用数据库提供的工具或第三方工具分析慢查询日志，找出执行最慢的查询语句。

- **MySQL**提供了`mysqldumpslow`工具来分析慢查询日志。

### 3. 使用EXPLAIN分析执行计划

对于发现的慢查询，使用数据库提供的EXPLAIN命令（或类似的工具）来查看SQL语句的执行计划，了解数据库如何执行这些查询，包括是否使用索引、如何进行表连接等。

### 4. 索引优化

根据执行计划，检查是否缺少索引或存在不合适的索引。添加或优化索引通常是提高查询性能的最快方法。但注意，不恰当的索引可能反而降低性能。



## 负载均衡算法有哪些？

1. **轮询算法**：将请求依次分配给每个服务器，循环进行；
2. **最小连接数算法**：将请求发送到当前连接数最少的服务器上。这样可以避免流量集中在少数几台服务器上，但无法考虑服务器的性能差异；
3. **最少响应时间**：将请求发送到响应时间最短的服务器上。通常需要服务器主动报告其响应时间，需要额外的监控和通信开销；
4. **哈希**：据请求的某些属性（如客户端IP地址、URL等）计算哈希值，然后将请求分配给特定的服务器。这种方法可以确保同一客户端的请求始终发送到同一台服务器，适用于需要维持会话状态的场景；
5. **加权轮询（Weighted Round Robin）**：与简单轮询类似，但可以给每个服务器分配一个权重，根据其性能或配置的不同，来分配不同的负载；
6. **加权最小连接数（Weighted Least Connections）**：与最小连接数类似，但考虑了服务器的权重，将请求发送到连接数最少且权重最高的服务器上；
7. **一致性哈希（Consistent Hashing）算法**：是负载均衡领域中非常重要的一种算法，它在分布式系统中有着广泛的应用。一致性哈希算法的主要思想是将请求的键（例如URL或其他标识符）通过哈希函数映射到一个固定范围的环形空间中，然后通过同样的哈希函数将服务器节点映射到环上。当一个请求到来时，通过一致性哈希算法确定该请求应该被路由到环上的哪个节点上，进而选择相应的服务器进行处理；
