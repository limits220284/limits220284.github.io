# Redis持久化

## RDB

### 方式

#### 1、save

执行了save命令，就会在主线程生成RDB文件，由于和执行操作命令在同一个线程，所以如果写入RDB文件的时间太长，就会阻塞主线程；

#### 2、bgsave

创建一个子进程来生成RDB文件，这样可以避免主线程的阻塞；



> 执行快照时，数据能被修改吗？

执行bgsave时候，redis仍然可以继续处理操作命令，也就是数据能够被修改的；

### 写时复制技术(COW)

1. 执行bgsave时，通过fork()创建子线程，此时创建的子线程和父线程共享同一片内存区域，因为创建子线程的时候，会复制父进程的页表，但是页表指向的物理内存还是一个；
   - 如果该块内存区域没有发生变化，直接进行拷贝即可；
   - 如果父进程要修改共享数据的某一块数据，就会发生写时复制，于是这块的数据的物理内存就会被复制一份(只有被修改的那块，其余的不变)，然后**主进程在数据副本**进行修改操作，子进程将原来的数据写入到RDB文件中；

### 优点

1. 保存的是全量的数据，加载速度较AOF快；

### 缺点

1. 如果频率太低，两次快照之间服务器发生宕机，就可能丢失比较多的数据；
2. 如果频率太高，造成额外的性能开销；



### tips

1. RDB文件加载工作实在服务器启动时自动执行的，Redis并没有提供专门加载RDB文件的命令；

2. 可以配置save时机

   ```shell
   save 900 1 # 900s之内，redis数据修改了1次
   save 300 10 # 300s之内，redis数据修改了10次
   save 60 10000 # 60s之内，redis数据修改了10000次
   ```



## AOF

### 概念

1. AOF持久化记录了Redis执行的所有写操作命令，并在Redis重启时，通过重新执行这些命令来重建数据
2. AOF默认关闭

如果需要开启AOF，需要在redis.conf中进行如下的配置；

```shell
appendonly yes # 表示是否开启AOF持久化
appendfilename "appendonly.aof" # AOF持久化文件的名称
```

### 过程

1. 客户端发送指令；
2. redis执行指令，写入内存；
3. 如果指令执行成功，redis记录命令追加到server.aof_buf缓冲区；
4. 通过write()系统调用，将缓冲区的内容数据写入到AOF文件中，位于page_cache中；
5. 通过写回策略，将page_cache中的内容写入到磁盘中；

### 写回策略

1. always：每次执行完指令，直接将AOF文件同步到磁盘中；
2. EverySec：写到AOF文件中后，每1s同步缓冲区的内容到磁盘中；
3. No：redis不控制从page_cache到磁盘的操作，由操作系统来决定；
4. 这三种策略都是通过fsync()系统调用进行操作的；



### AOF重写机制

重写机制的提出，主要是为了解决AOF文件太大的问题；

当文件大小超过阈值，就会触发重写机制，来压缩AOF文件；

**具体做法**：通过读取key最新的value，然后用一条新的命令记录，写到AOF文件中即可，之前的旧命令直接删除；**一般重写AOF文件，需要新建一个文件，避免因为重写失败，造成的数据污染**；



### 优点

1. 数据安全性高，因为AOF是实时记录写操作，在Redis宕机时，不会出现数据丢失

### 缺点

1. AOF记录信息很多，文件通常很大
2. 重建数据时，加载数据比较慢





## 混合持久化

提出的原因：解决AOF和RDB各自的优缺点；

如果想要开启混合持久化功能，可以在 Redis 配置文件将下面这个配置项设置成 yes：

```shell
aof-use-rdb-preamble yes
```



#### 过程

1. AOF重写日志的时候，fork出来的重写子进程会先将主进程共享的内存数据以RDB的方式存到AOF文件中；
2. 这个过程中的操作命令记录在重写缓冲区中；
3. 重写缓冲区的增量命令以AOF的方式添加到AOF文件中；
