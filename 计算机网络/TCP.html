TCP ​

重传机制 ​

1.超时重传 ​

RTT：Round-Trip Time，往返时延

RTO：Retransmission Timeout 超时重传时间

超时触发重传存在的问题是，超时周期可能相对较长。那是不是可以有更快的方式呢？

2. 快速重传 ​

不以时间为驱动，而是以数据驱动重传。

假如发送的是数据1-6，其中2丢失了，服务端在接收到三个同样的ACK时候，就会触发重发机制

快速重传机制只解决了一个问题，就是超时时间的问题，但是它依然面临着另外一个问题。就是重传的时候，是重传一个，还是重传所有的问题。 比如2，3都丢了，但是接收到4，5，6的时候会返回三次ACK 2，发送方无法确定重传的个数是多少；

3. SACK 方法 ​

SACK(selective acknowledgment)，选择性确认

  1. 需要在头部加入一个SACK的东西，它可以将已经收到的数据的信息发送给发送方，这样发送方就可以知道哪些数据收到了，哪些数据没有收到，知道了这些信息，就可以只重传丢失的数据；

相当于接收方会通过SACK返回收到部分数据的信息；

但是这样也会存在一些问题：

  1. 如果是因为网络原因造成的发送的信息很久未到，可以让发送端知道原因；
  2. 如果是因为ACK丢包了，也可以告诉发送端，不是因为自己的原因，自己发送的已经到达了；

Duplicate SACK 又称 D-SACK，其主要使用了 SACK 来告诉「发送方」有哪些数据被重复接收了。

用来解决上面的两个问题:

  1. 如果是因为ACK报文丢失了，下次发送方再次发送该报文的时候，接收方会在SACK报文中携带这段重复接受数据的信息；

滑动窗口 ​

存在原因：传统的TCP都是发送完毕数据之后，进行一个ACK，再发送下一个，这样效率很低；

进而TCP引入了窗口这个概念，有了窗口，可以指定窗口大小，无需等待确认应答，而可以继续发送数据的最大值；

就算某个时刻，ack = 600丢失了，发送方也可以通过ACK = 700来确定之前的数据都被接收到了；称之为累计应答；

窗口大小由哪一方决定？

TCP头部存在一个字段Window，这个字段是接收端告诉发送端还有多少缓冲区可以接收数据。于是发送端就可以根据这个接收端的处理能力来发送数据，而不会导致接收端处理不过来；

所以通常是由接收方窗口大小来决定的；

发送方如何表示窗口？

  1. SND.WND：表示发送窗口的大小；
  2. SND.UNA： 是一个绝对指针，指向的是已经发送但是未收到确认的第一个字节的序列号；
  3. SND.NXT：是一个绝对指针，指向未发送但是可以发送的第一个字节的序列号；

接收方如何表示窗口？

RCV.WND：表示接收窗口的大小

RCV.NXT：表示期望从发送方发送来的下一个数据字节的序列号

未收到并且不可以接受的数据序列号就可以通过RCV.NXT + RCV.WND计算出来;

流量控制 ​

存在原因：发送方不能无脑的发送数据给接收方，要考虑接收方的处理能力，如果一直无脑的发数据给对方，但是对方处理不过来，就会导致触发重发机制，从而导致网络流量的无端浪费；

怎么做的？

  1. 发送方窗口大小，控制发送的频率和大小
  2. 通过操作系统的缓冲区大小，来动态维护窗口大小；

拥塞控制 ​

存在原因：主要为了解决网络堵塞问题，流量控制只是为了解决发送/接收双方的信息处理能力；

在网络出现拥堵时，如果继续发送大量数据包，可能会导致数据包时延、丢失等，这时 TCP 就会重传数据，但是一重传就会导致网络的负担更重，于是会导致更大的延迟以及更多的丢包，这个情况就会进入恶性循环被不断地放大....

拥塞窗口？和发送窗口什么关系？

拥塞窗口cwnd是发送方维护的一个状态变量，它会根据网络的拥塞程度动态变化；

swnd(发送窗口) = min(cwnd(拥塞窗口)，rwnd(接收窗口))

如果没有发生堵塞，cwnd就会增大，否则减小

如何得知网络是否出现了堵塞？

很简单，只要发送方没有在规定时间内接收到ACK应答报文，就是发生了超时重传，就会认为网络出现了拥塞！

拥塞控制存在哪些算法?

  1. 慢启动
  2. 拥塞避免
  3. 拥塞发生
  4. 快速恢复

慢启动 ​

规则：当发送方每收到一个ACK，拥塞窗口cwnd的大小就会加1；

  • 连接建立完成后，一开始初始化 cwnd = 1，表示可以传一个 MSS 大小的数据。
  • 当收到一个 ACK 确认应答后，cwnd 增加 1，于是一次能够发送 2 个
  • 当收到 2 个的 ACK 确认应答后， cwnd 增加 2，于是就可以比之前多发2 个，所以这一次能够发送 4 个
  • 当这 4 个的 ACK 确认到来的时候，每个确认 cwnd 增加 1， 4 个确认 cwnd 增加 4，于是就可以比之前多发 4 个，所以这一次能够发送 8 个。

难道可以一直增加吗？

参数ssthresh(slow start threshold) 状态变量

  • 当 cwnd < ssthresh：慢启动
  • cwnd >= ssthrest：拥塞避免算法

拥塞避免算法 ​

规则：每当收到一个ack后，cwnd增加 $\frac{1}{cwnd} $

接上前面的慢启动的栗子，现假定 ssthresh 为 8：

  • 当 8 个 ACK 应答确认到来时，每个确认增加 1/8，8 个 ACK 确认 cwnd 一共增加 1，于是这一次能够发送 9 个 MSS 大小的数据，变成了线性增长。

但是这样还会出现一直增长的情况，于是就会出现丢包情况，这时就需要对丢掉的数据包进行重传，

当触发了重传机制，也就进入了拥塞发生算法;

拥塞发生算法 ​

存在原因：拥塞避免算法随着时间增长，数据包过多，导致的网络堵塞，造成了重传机制；

  • 超时重传
  • 快速重传

发生超时重传的拥塞发生算法

  • ssthresh设为cwnd/2
  • cwnd重置为1(恢复初始值)

怎么查看系统的cwnd初始化值？

shell
ss -nli | grep cwnd

但是这样问题在于，之前的发送大小一般很大，但是突然恢复到初始值，比较激进，容易造成卡顿；

发生快速重传的拥塞发生算法

还有更好的方式，前面我们讲过「快速重传算法」。当接收方发现丢了一个中间包的时候，发送三次前一个包的 ACK，于是发送端就会快速地重传，不必等待超时再重传。

TCP 认为这种情况不严重，因为大部分没丢，只丢了一小部分，则 ssthresh 和 cwnd 变化如下：

  • cwnd = cwnd/2 ，也就是设置为原来的一半;
  • ssthresh = cwnd;
  • 进入快速恢复算法

快速恢复算法 ​

一般和快速重传算法结合使用，在进入快速恢复之前，cwnd和ssthresh已经被更新了

  • cwnd = cwnd/2 ，也就是设置为原来的一半;
  • ssthresh = cwnd;

步骤：

  1. 拥塞窗口 cwnd = ssthresh + x(代表有三个数据包被收到了)；
    • 相当于失去的三个数据包都收到了，然后处于拥塞避免状态，那么就直接+ 1/cwnd即可；
  2. 重传丢失的数据包；
  3. 如果再重复收到ACK，那么cwnd + 1；
    • 为了快速重传丢失的包，只能增大窗口
  4. 如果收到新数据的ACK后，把cwnd设置为第一步的ssthresh值，原因是该ACK确认了新的数据，说明从duplicated ACK时候的数据都已经收到，该恢复过程已经结束，可以回到之前的状态了，也即再次进入拥塞避免状态；
    • 收到新数据ACK，表示收到了新数据，此时没必要保持这种状态了，因为收到重复ACK是为了快速重发，并没有进行拥塞避免；

首先，快速恢复是拥塞发生后慢启动的优化，其首要目的仍然是降低 cwnd 来减缓拥塞，所以必然会出现 cwnd 从大到小的改变。

其次，过程2（cwnd逐渐加1）的存在是为了尽快将丢失的数据包发给目标，从而解决拥塞的根本问题（三次相同的 ACK 导致的快速重传），所以这一过程中 cwnd 反而是逐渐增大的。

TCP Keepalive机制的工作原理 ​

  1. 启用Keepalive：在大多数操作系统中，TCP Keepalive默认可能是关闭的或者具有一个非常长的超时时间。应用程序可以通过设置socket选项来启用Keepalive，并配置相关参数，如探测间隔和探测次数。
  2. 探测（Probe）发送：一旦Keepalive被启用，并且在配置的超时时间内没有在连接上检测到任何数据交换，操作系统将开始发送Keepalive探测。这些探测是特殊的TCP包，目的是检查对端是否仍然响应。
  3. 响应检查：如果对端正常响应了Keepalive探测，表示连接仍然有效，操作系统将重置Keepalive计时器，并在下一个周期再次等待。如果对端没有响应，操作系统会继续发送探测，直到达到配置的探测次数限制。
  4. 连接终止：如果所有Keepalive探测都没有得到响应，操作系统将认为连接已经断开，并通知应用程序。这通常通过在socket上报告一个错误来实现。

怎么解决TCP粘包问题？ ​

1、固定消息长度 ​

和UDP操作方式相同，规定每个发送的消息的长度都是一致的，当接收方接收到这个长度的消息的时候，就认为这是一个有效的消息；

缺点：不够灵活，实际不常用

2、特殊字符当作边界 ​

可以在两个用户消息之间插入一个特殊的字符串，这样接收方在接收数据时，读到了这个字符串，就代表已经读完了一个完整的消息；

HTTP就是一个非常好的例子;

HTTP 通过设置回车符、换行符作为 HTTP 报文协议的边界。

有一点要注意，这个作为边界点的特殊字符，如果刚好消息内容里有这个特殊字符，我们要对这个字符转义，避免被接收方当作消息的边界点而解析到无效的数据

3、自定义消息结构 ​

我们可以自定义一个消息结构，由包头和数据组成，其中包头是固定大小的，包头里面有一个字段可以指定后续数据有多大；

c++
struct { 
    u_int32_t message_length; 
    char message_data[]; 
} message;
