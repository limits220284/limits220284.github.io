# HTTP



## HTTP状态码?

200:请求成功：服务器已经成功处理了请求；

201:资源已创建：请求成功，并且服务器创建了新的资源；

202:已接受：请求已接受，但是尚未处理：



301：永久重定向：请求网页已经永久移动到新的位置；

302：临时重定向：说明请求的资源还在，但暂时需要用另一个URL来访问；

304：未修改：自从上次请求后，请求的网页未修改过，服务器返回此响应时，不会返回网页内容，重定向到已经存在的缓存文件中；



400：请求错误：入参有问题

401：未认证：请求要身份认证，对于需要登陆的网页，服务器返回此响应；

402：需要付费

403：禁止访问：服务器拒绝请求；

404：页面没找到：Not Found；

405：请求资源方式不允许：Post请求了一个Get接口；



500：服务器内部错误：服务里内部逻辑存在问题；

501：尚未实施：服务器不具备完成请求的功能；

502：错误网关：服务器作为网关或者代理，从上游服务器收到无效响应；

503：服务不可用：停机维护，暂时的状态；

504：网关超时：服务器作为网关代理，但是没有及时从上游服务器收到请求；





##  HTTPS

1. 防止监听，采用对称密钥进行加密
2. 防止篡改，采用数字签名进行认证
3. 防止冒充，采用CA作为认证
4. 客户端要和服务端进行连接时，首先会通过非对称密钥协商一个对称加密密钥，后续通过对称密钥进行通信;
5. 如果这个过程中因为自己的原因，点进入了一个非合法的网站，那么就有可能存在冒充的风险;

## HTTP/1.1 相比于HTTP/1.0 提高了什么性能?

### 优点:

1. 使用长连接改善了1.0短链接造成的额外开销
2. 支持管道网络传输，只要第一个请求发出去了，不必等其回来，就可以发出第二个请求出去

### 缺点

1. 请求/响应头部未经压缩就发送，首部信息越多，延迟越大
2. 服务器是按请求的顺序响应的，如果服务器响应慢，就会导致客户端一直请求不到数据，也就是队头阻塞；
3. 没有请求优先级控制
4. 请求只能通过客户端开始，服务器端只能被动响应

## HTTP2做了什么优化？(相比于1.1)

### 优点

1. 头部压缩
   - HPACK算法，如果同时发送多个请求，头部相似，协议会帮你消除相似的部分
   - 在客户端和服务器同时维护一张头信息表，所有字段都会存入这个表，生成一个索引号，以后就不发送同样字段了，只发送索引号，这样就**提高速度**了
2. 二进制格式
   - 不采用1.1中的纯文本形式报文，全面采用了二进制格式，分为**头部信息帧**和**数据帧**
3. 并发传输
   - 引入了Stream概念，一个TCP连接上可以包含多个Stream，Message包含一条或者多个Frame，每一个响应对应的Stream ID是独一无二的，客户端会根据相同的Stream ID组装消息
4. 服务器主动推送资源
   - 客户端和服务器**双方都可以建立 Stream**， Stream ID 也是有区别的，客户端建立的 Stream 必须是奇数号，而服务器建立的 Stream 必须是偶数号。
   - 在 HTTP/2 中，客户端在访问 HTML 时，服务器可以直接主动推送 CSS 文件，减少了消息传递的次数。

### 缺点

1. **HTTP/2 是基于 TCP 协议来传输数据的，TCP 是字节流协议，TCP 层必须保证收到的字节数据是完整且连续的，这样内核才会将缓冲区里的数据返回给 HTTP 应用，那么当「前 1 个字节数据」没有到达时，后收到的字节数据只能存放在内核缓冲区里，只有等到这 1 个字节数据到达时，HTTP/2 应用层才能从内核中拿到数据，这就是 HTTP/2 队头阻塞问题。**

## HTTP3 做了哪些优化？

### 优点

1. HTTP/2 队头阻塞的问题是因为 TCP，所以 **HTTP/3 把 HTTP 下层的 TCP 协议改成了 UDP！**
2. UDP 发送是不管顺序，也不管丢包的，所以不会出现像 HTTP/2 队头阻塞的问题。大家都知道 UDP 是不可靠传输的，但基于 UDP 的 **QUIC 协议** 可以实现类似 TCP 的可靠性传输。QUIC 有以下 3 个特点。
   - 无队头阻塞
   - 更快的连接建立
     - 对于 HTTP/1 和 HTTP/2 协议，TCP 和 TLS 是分层的，分别属于内核实现的传输层、openssl 库实现的表示层，因此它们难以合并在一起，需要分批次来握手，先 TCP 握手，再 TLS 握手。
     ![TCP HTTPS（TLS/1.3） 和 QUIC HTTPS ](https://cdn.xiaolincoding.com/gh/xiaolincoder/ImageHost/%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BD%91%E7%BB%9C/HTTP/28-HTTP3%E4%BA%A4%E4%BA%92%E6%AC%A1%E6%95%B0.png)
   - 连接迁移
     -  QUIC 是一个在 UDP 之上的**伪** TCP + TLS + HTTP/2 的多路复用的协议。
     - 而 QUIC 协议没有用四元组的方式来“绑定”连接，而是通过**连接 ID** 来标记通信的两个端点，客户端和服务器可以各自选择一组 ID 来标记自己，因此即使移动设备的网络变化后，导致 IP 地址变化了，只要仍保有上下文信息（比如连接 ID、TLS 密钥等），就可以“无缝”地复用原连接，消除重连的成本，没有丝毫卡顿感，达到了**连接迁移**的功能。





### 缺点

1. QUIC 是新协议，对于很多网络设备，根本不知道什么是 QUIC，只会当做 UDP，这样会出现新的问题，因为有的网络设备是会丢掉 UDP 包的，而 QUIC 是基于 UDP 实现的，那么如果网络设备无法识别这个是 QUIC 包，那么就会当作 UDP包，然后被丢弃。
2. HTTP/3 现在普及的进度非常的缓慢，不知道未来 UDP 是否能够逆袭 TCP。
